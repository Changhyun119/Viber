# 디자인 시스템 방향

## 1. 디자인 목표

이 서비스는 "개발자 게시판"보다 "프로덕트 쇼케이스"에 가까워야 한다.

### 목표 요약

- 카드 중심
- 미디어 우선
- 텍스트는 짧고 명확하게
- 빠른 체험 유도
- 높은 정보 밀도, 낮은 혼잡도

## 2. 전체 톤

- 포럼형 텍스트 덩어리보다 제품 소개 페이지에 가깝게 간다.
- 기술 스택 자랑보다 사용자 가치가 먼저 보여야 한다.
- "무엇을 만들었는지"와 "바로 써볼 수 있는지"가 2초 안에 읽혀야 한다.

## 3. 레이아웃 원칙

### 홈/탐색

- 데스크톱: 3열 카드 그리드
- 태블릿: 2열
- 모바일: 1열

### 상세 페이지

- 첫 화면에 media + CTA가 들어와야 한다.
- 모바일에서는 `Try` 버튼을 하단 sticky로 노출하는 방식을 고려한다.

## 4. 카드 설계 원칙

프로젝트 카드는 이 서비스의 핵심 컴포넌트다.

### 기본 구조

- 상단: 대표 이미지 16:10
- 제목 + 상태 배지
- 한 줄 소개 2줄 clamp
- 핵심 배지 최대 3개
- maker alias
- 최근 활동일
- 댓글 수 / 저장 수
- `Try`
- `Details`

### 표시 규칙

- 배지는 최대 3개만 노출
- 설명은 2줄 이상 자르기
- 버튼은 최대 2개
- GitHub는 카드보다 상세에서 더 강조

## 5. 카드 타입

- 기본 카드
- 피처드 카드
- 업데이트 카드
- 피드백 카드

각 카드 타입은 구조는 유지하되 라벨과 강조 정도만 다르게 가져간다.

## 6. 배지 설계 원칙

배지는 정보 전달용으로 최소화한다.

### 우선 후보

- Web
- Mobile
- Open Source
- No Signup
- Free
- Beta

### 원칙

- 시각 장식이 아니라 판단 보조 정보여야 한다.
- 과도하게 많아지면 카드 읽기 속도가 느려진다.

## 7. CTA 우선순위

### 1순위

- `Try`

### 2순위

- `Details`

### 보조

- `Save`
- `Share`
- `GitHub`

핵심은 "읽게 만드는 것"이 아니라 "써보게 만드는 것"이다.

## 8. 상세 페이지 UX 원칙

- 프로젝트명과 가치 제안이 먼저 보여야 한다.
- 미디어 없이 긴 문단부터 나오면 안 된다.
- 활동 히스토리는 타임라인 감각이 살아 있어야 한다.
- 댓글 영역은 제품 피드백 공간처럼 보여야 한다.

## 9. 입력 UX 원칙

- 긴 단일 폼보다 단계형 마법사가 적합하다.
- 각 단계는 왜 이 정보를 입력하는지 명확해야 한다.
- 마지막에는 실제 카드/상세 미리보기를 보여준다.

## 10. 빈 상태와 오류 상태

이 서비스는 목록 화면이 많기 때문에 상태 설계가 중요하다.

### 빈 상태에서 전달해야 할 것

- 아직 프로젝트가 없음
- 첫 프로젝트 등록 유도
- 필터가 너무 강한 경우 필터 초기화 안내

### 오류 상태에서 전달해야 할 것

- 다시 시도
- 홈으로 이동
- 신고 또는 문의 경로

## 11. 반응형 원칙

- 모바일에서도 카드 의미가 유지되어야 한다.
- 필터는 접히되 사라지면 안 된다.
- 상세의 핵심 CTA는 스크롤 중에도 접근 가능해야 한다.

## 12. 컴포넌트 우선순위

초기 컴포넌트 설계 우선순위는 아래와 같다.

- ProjectCard
- FilterChipBar
- FeaturedProjectCard
- ActivityFeed
- CommentThread
- SubmissionWizard
- ReportDialog
- VerificationBadge

## 13. 시각적 실패 조건

아래 상태는 디자인 실패로 본다.

- 텍스트 게시판처럼 보임
- 카드보다 설명문이 먼저 읽힘
- `Try`가 눈에 띄지 않음
- 프로젝트보다 계정 요소가 더 크게 보임
- 상태와 배지가 너무 많아 혼잡함
