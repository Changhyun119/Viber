# 랭킹, 운영, 검수 정책

## 1. 랭킹 원칙

메인 랭킹은 단순 인기 투표가 아니라 "실제 사용 유도"와 "실제 반응"을 반영해야 한다.

### 핵심 신호

- 고유 outbound click
- 저장 수
- 댓글 수
- 최근성
- 클릭 대비 반응률

### 하지 않는 것

- 좋아요 총량만으로 정렬
- 클릭 수만 높고 반응이 없는 프로젝트 과대 노출
- 피처드 프로젝트와 일반 랭킹 혼합

## 2. 트렌딩 로직 가이드

정확한 수치 가중치는 운영 중 조정하되, 방향은 아래를 고정한다.

- 동일 세션 또는 동일 사용자의 짧은 시간 내 중복 클릭은 1회만 반영한다.
- 저장과 댓글이 전혀 없는 클릭만 많은 프로젝트는 과도하게 올리지 않는다.
- 최신성 보정이 필요하다.
- 관리자 피처드는 별도 섹션으로 분리한다.

## 3. 노출 면 구성

### 홈

- 추천 프로젝트
- 새로 나온 프로젝트
- 피드백 기다리는 프로젝트
- 최근 업데이트

### 탐색

- 전체 검색 결과
- 필터별 목록
- 정렬 결과

### 상세

- 관련 프로젝트

## 4. 등록 즉시 차단 또는 보류 조건

- 라이브 링크가 죽어 있음
- 단순 랜딩페이지만 있고 제품 체험이 안 됨
- 남의 프로젝트 복붙
- affiliate 또는 광고성 링크만 존재
- 과도한 외부 링크
- 혐오, 불법, 성인물
- 동일 URL 중복 제출
- 설명이 지나치게 빈약함

## 5. 필수 안티스팸 장치

- CAPTCHA 또는 Turnstile
- IP 기반 rate limit
- 동일 도메인 중복 감지
- 금칙어 필터
- 신고 기능
- 관리자 운영 큐
- 링크 상태 주기적 확인

## 6. 링크 헬스체크 정책

### 기본 규칙

- `live_url`은 주기적으로 상태 확인
- 3회 이상 실패 시 `broken link` 경고 표시
- 장기간 죽어 있으면 `archived` 또는 `inactive` 후보

### 운영 목적

- 탐색 품질 유지
- 허수 프로젝트 제거
- 오래된 프로젝트 정리

## 7. 중복 감지 정책

중복은 아래 기준으로 탐지한다.

- 정규화된 `live_url`
- 정규화된 `github_url`
- 유사한 제목 + 동일 도메인

완전 자동 삭제보다는 관리자 큐로 보내는 방식이 안전하다.

## 8. 신고 정책

### 신고 대상

- 프로젝트
- 활동
- 댓글

### 신고 사유

- 스팸
- 중복
- 사칭
- 죽은 링크
- 부적절한 콘텐츠
- 허위 또는 오해 소지

### 관리자 액션

- publish
- reject
- hide
- limit
- archive
- feature

모든 조치는 `moderation_actions`에 기록한다.

## 9. 제출/운영 상태 흐름

기본 제출/운영 흐름은 아래처럼 본다.

1. 제출
2. 자동 규칙 검사
3. 소유권 확인
4. 즉시 게시 또는 자동 차단/보류
5. 운영 이력 저장

## 10. 운영 도구 최소 요구사항

- 운영 이슈 큐 필터
- 신고 사유별 정렬
- 중복 후보 비교
- 링크 상태 표시
- 프로젝트 상태 변경
- 피처드 편성

## 11. 핵심 운영 이벤트

초기 분석 이벤트는 아래 정도면 충분하다.

- `project_card_impression`
- `project_try_click`
- `project_detail_view`
- `project_save`
- `comment_created`
- `project_submitted`
- `project_published`
- `report_created`

## 12. 운영 관점 주의점

- 메이커만 모이는 폐쇄적 공간이 되면 실제 사용자 피드백이 약해진다.
- 익명 도배를 방치하면 품질보다 소음이 커진다.
- 자유게시판을 너무 빨리 열면 프로젝트 허브라는 정체성이 무너진다.
- 메인 랭킹을 너무 단순화하면 조작에 취약해진다.

## 13. 오픈 초기 운영 권장안

- 시드 프로젝트 20~30개 선등록
- 메인 추천용 프로젝트 5개 준비
- 피드백 요청 예시 3개 준비
- 업데이트 예시 5개 준비
- 신고 처리 가이드 먼저 문서화

## 14. 운영 품질 판단 기준

- 실제로 눌러볼 수 없는 프로젝트 비율이 낮아야 한다.
- 중복 프로젝트 노출이 적어야 한다.
- dead link 비율이 낮아야 한다.
- 첫 피드백 발생 시간이 짧아야 한다.
